# PorySuite-Z — Claude Context for New Threads

## What is this project?
PorySuite-Z is a unified editor for pokefirered decomp ROM hacking projects. It combines two previously separate tools:
- **PorySuite** — edits Pokemon data (species, items, moves, trainers, starters, pokedex, config)
- **EVENTide** — edits maps, scripts, events, layouts, region maps, and text

- **.s File Overwrite Protection (2026-04-08) — COMPLETE**: pokefirered's Makefile has `%.s: %.mid $(MID_CFG_PATH)` — if midi.cfg OR any .mid is newer than the .s, mid2agb regenerates the .s from the .mid. For tool-edited songs the .mid is a 26-byte placeholder, so mid2agb produces empty garbage that wipes the real music. Two-layer protection: (1) `write_midi_cfg()` touches ALL .s files after rewriting midi.cfg so they're newer. (2) Every .s-writing path (`save_song_file`, MIDI import, .s import, song replace) backdates the corresponding .mid by 2 seconds. This is a **critical invariant** — any code that writes midi.cfg or .s files MUST maintain timestamp ordering (.s newer than both .mid and midi.cfg). Files: `core/sound/song_table_manager.py`, `core/sound/song_writer.py`, `ui/dialogs/midi_import_dialog.py`, `ui/dialogs/s_file_import_dialog.py`, `ui/sound_editor_tab.py`.

## Critical: .s File Timestamp Protection

pokefirered's Makefile rule `%.s: %.mid $(MID_CFG_PATH)` means midi.cfg is a **dependency for EVERY .s file**. If midi.cfg is rewritten (by song delete, rename, import, cleanup), ALL .s files become stale and `make` will re-run mid2agb on every song. For tool-edited songs where the .mid is a 26-byte placeholder, mid2agb produces empty/garbage output that **destroys the real music**.

**Invariant that MUST be maintained:**
- `.s` files must ALWAYS be newer than both their `.mid` AND `midi.cfg`
- `write_midi_cfg()` touches all .s files after full rewrite
- `register_song()` in `midi_importer.py` touches all .s files after appending to midi.cfg (was the root cause of recurring mus_evil wipes — this path was unprotected until 2026-04-09)
- Every .s write path backdates the corresponding .mid by 2 seconds
- `_on_make()` in `unified_mainwindow.py` touches all .s files before every build (defense-in-depth)
- **Any new code that writes midi.cfg, .s files, or .mid files MUST maintain this timestamp ordering**

Similarly, editing instrument .bin files (loop settings, sample replacement) writes directly to the binary on disk. This is safe because .bin files are `.incbin`'d by `direct_sound_data.inc` — they're not part of the midi.cfg dependency chain.
